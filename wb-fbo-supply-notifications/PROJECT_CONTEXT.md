# Контекст проекта: уведомления по поставкам FBO WB

Этот файл нужен как стартовый контекст для нового чата без истории переписки. В нем описано, что делает подпроект, какие источники данных используются, как формируются сообщение и Markdown-файл, как настроена автоматизация и куда вносить изменения.

## Назначение

Подпроект `wb-fbo-supply-notifications` собирает ежедневную сводку по поставкам FBO Wildberries и отправляет ее в Пачку:

- текстовое сообщение с товарами, требующими действий;
- Markdown-файл с таблицами и полным списком поставок;
- отдельное сообщение в тред, если у товаров из текстовой сводки не указан ответственный маркетолог.

Основная бизнес-цель: быстро увидеть товары, по которым скоро/только что пришла поставка FBO, но до прихода был низкий остаток, и проверить цену, БЗО и рекламу.

Секреты не хранятся в репозитории. Для GitHub Actions нужны repository secrets:

- `PACHCA_CHAT_ID` - id чата Пачки, куда отправляется сообщение.

Дополнительная переменная:

- `abcage-notify-parser@notification-abcage.iam.gserviceaccount.com`

Ему должен быть выдан доступ читателя ко всем Google Sheets, которые читает отчет.

## Автоматизация

Текущая автоматизация работает через GitHub Actions:

- расписание: каждый день в `08:00 Asia/Tbilisi`;
- cron в UTC: `0 4 * * *`;
- ручной запуск: `workflow_dispatch`;
- workflow сам собирает отчет, отправляет сообщение и Markdown-файл в Пачку, затем сохраняет артефакт в GitHub Actions.

В workflow включен `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24=true`, чтобы стандартные GitHub Actions запускались на Node 24 runtime.

Важный нюанс: GitHub scheduled workflows не гарантируют запуск секунда-в-секунду. GitHub может задержать schedule при высокой нагрузке, особенно в начале часа. Если нужна более надежная cron-инфраструктура, проект можно переносить в Cloudflare Workers, но текущий Python-код нельзя перенести туда без портирования, потому что он использует subprocess и `npx mcp-remote`.

## Источники данных

### 1. Принятые поставки за вчера и сегодня

Для блока `ВЧЕРА и СЕГОДНЯ` данные берутся не из Google Sheets, а из ABCAGE Analyzer MCP через SQL:

- `mp.wb_core__supply`
- `mp.wb_core__supply_contents`

Логика:

- период: `START - 1 день` включительно до `START + 1 день` не включительно;
- `START` считается по `REPORT_TZ`;
- количество поставки: сумма `count_fact`;
- группировка: по `sku`;
- в отчетный день эти строки кладутся как дата `START`;
- поставки за вчера и сегодня объединяются в один список без отдельного деления по датам.

### 2. Будущие поставки

Для будущих дней данные берутся из Google Sheets:

- spreadsheet id: `1kLX5hGPK3g8HRno39POiHheg9UIFXNkKQGKcpAXX1KM`
- вкладка `ВБ. Новый`, range `'ВБ. Новый'!C:O`
- вкладка `ВБ. Регионы`, range `'ВБ. Регионы'!C:O`

Используемые колонки внутри диапазона `C:O`:

- `C` - дата поставки;
- `L` - строка с информацией о товаре;
- `M` - внутренний/поставочный артикул;
- `N` - barcode;
- `O` - количество к отгрузке.

Если один и тот же артикул встречается в двух вкладках или несколько раз в одной вкладке, поставки суммируются по артикулу и дате.

### 3. Карточка товара, название и категория

Через ABCAGE Analyzer MCP:

- `mp.wb_core__card`

Используется для сопоставления barcode/WB артикула, названия (`short_name`) и категории (`object`).

### 4. Текущий остаток FBO

Через ABCAGE Analyzer MCP:

- `mp.mp_core__realtime_stocks_data`

Берется последняя доступная дата:

```sql
date = (SELECT MAX(date) FROM mp.mp_core__realtime_stocks_data)
```

Остаток: сумма `fbo_real` по WB артикулу.

### 5. РК, статусы и траты

Наличие неархивной РК определяется через:

- `mp.wb_core__campaign_card`
- `mp.wb_core__campaign`
- fallback по `campaign.sku_list`

В РК считаются кампании со state:

- `4`
- `9`
- `11`

Траты за 3 дня берутся из:

- `mp.wb_core__campaign_stat_daily_sku`

Период трат:

- `START - 2 дня`;
- `START - 1 день`;
- `START`;

то есть `date_at BETWEEN START - 2 AND START`.

### 6. Заказы за 7 дней

Через ABCAGE Analyzer MCP:

- `mp.wb_core__order`

Период:

- `START - 6 дней` по `START` включительно.

Учитываются только неотмененные заказы:

```sql
COALESCE(is_cancel, 0) = 0
```

Подсчет:

```sql
COUNT(DISTINCT COALESCE(NULLIF(srid, ''), order_number))
```

### 7. Цена и СПП

Через ABCAGE Analyzer MCP:

- основной источник: `mp.mp_core__realtime_prices`;
- fallback: `mp.wb_core__price`.

В отчет выводятся:

- текущая цена до СПП;
- цена с СПП;
- процент СПП.

Важно: СПП устанавливает Wildberries. Мы его не контролируем. Управлять можно только ценой до СПП, но ориентиры в сообщении даются по паре `цена / спп`.

### 8. БЗО, отзывы и рейтинг

Данные берутся из публичного карточного API Wildberries:

```text
GET https://card.wb.ru/cards/v4/detail
```

Параметры:

```text
appType=1
curr=rub
dest=-1257786
spp=30
ab_testing=false
nm=<WB артикула через ;>
```

Поля:

- `products[].feedbackPoints` -> БЗО;
- `products[].nmFeedbacks` -> количество отзывов;
- `products[].nmReviewRating` -> рейтинг.

Если `feedbackPoints > 0`, в Markdown-файле БЗО отображается как `да (<сумма>)`. Если `feedbackPoints` нет, `0` или карточка не вернулась, БЗО отображается как `нет`.

### 9. Маркетолог

Google Sheets:

- spreadsheet id: `1STPnPgj8xSrvN-F3K96bDj_pmunCICHTjaj358pRaB4`
- gid: `1574673852`
- range: `D1:H430`

Логика:

- колонка `D` - WB артикул;
- колонка `H` - тег маркетолога;
- в отчет попадает тег из `H`.

Если маркетолог не найден:

- в сообщении выводится `` `не указан маркетолог` ``;
- в Markdown-файле в колонке `Маркетолог` остается `-`;
- в конце Markdown-файла и в тред Пачки формируется запрос к `@a.nekrasov` на добавление ответственных маркетологов.

### 10. Менеджер

- gid: `586703293`
- range: `C1:F3492`

Логика:

- колонка `C` - WB артикул;
- колонка `F` - имя менеджера;
- маппинг имен в теги:


## Логика попадания в текстовую сводку

Markdown-файл содержит полный список по источникам, но текстовое сообщение в Пачку содержит только товары под условия действия.

### Блок `ВЧЕРА и СЕГОДНЯ`

Источник: принятые WB товары за вчера и сегодня по `count_fact`.

Условие попадания:

```text
qty > 0
и
текущий FBO - принятое количество <= 100
```

Это важно: текущий FBO может стать больше 100 после приемки, поэтому для вчера/сегодня проверяется остаток до прихода.

Формат поставки в сообщении:

```text
<текущий FBO> (+<принятое количество>)
```

Пример:

```text
333 (+336) / `679796751` / Коврик детский 2.3 складной 150x200 город
```

### Блок `ЗАВТРА`

Источник: Google Sheets по будущим поставкам.

Условие попадания:

```text
qty > 0
и
текущий FBO < 100
```

Формат поставки в сообщении:

```text
<текущий FBO> -> +<плановая поставка>
```

### Послезавтра

Послезавтра выводится в полном Markdown-файле, но сейчас не попадает в текстовую сводку Пачки. Текстовая сводка формируется только для:

- `ВЧЕРА и СЕГОДНЯ`;
- `ЗАВТРА`.

## Логика рекомендаций

В текстовом сообщении вложенные строки начинаются с символа `↳`. Не использовать вложенные Markdown-списки, табы или цитаты: Пачка визуально не сохраняет нормальную вложенность. Оставлен простой формат с `↳`.

Если у товара нет рекомендаций, строки `↳` не выводятся.

Последовательность рекомендаций в сообщении:

1. Цена.
2. БЗО.
3. РК.

### Цена

Условие:

```text
цена до СПП > 80 000 ₽
```

В сообщении:

```text
↳ ЦЕНА: **ПРОВЕРИТЬ ЦЕНУ** (*цена / спп: 1 060 000 ₽ / -*) / @manager
```

В Markdown-таблице действие пишется без капса:

```text
Цена: проверить цену
```

### БЗО

Условие:

```text
feedbackPoints <= 0
и
отзывов < 10
и
заказов за 7 дней < 10
```

В сообщении:

```text
↳ БЗО: **ВКЛЮЧИТЬ БЗО** (*отзывы: 0 (0 ★)*) / @manager
```

В Markdown-файле в колонке `БЗО` не пишется `ВКЛЮЧИТЬ БЗО`; там остается фактическое значение `нет` или `да (<сумма>)`. Рекомендация находится в колонке `Действие`.

### РК: создать

Условие:

```text
нет неархивной РК
```

В сообщении:

```text
↳ РК: **СОЗДАТЬ РК** (*траты 3д: 0 ₽*) / @marketer
```

Если маркетолог не указан:

```text
↳ РК: **СОЗДАТЬ РК** (*траты 3д: 0 ₽*) / `не указан маркетолог`
```

### РК: проверить активность

Условие:

```text
РК существует
и
траты за 3 дня < 3 000 ₽
и
блок = ВЧЕРА и СЕГОДНЯ
```

Для блока `ЗАВТРА` действие `ПРОВЕРИТЬ АКТИВНОСТЬ РК` не выводится. Для завтра РК-рекомендация выводится только если кампании нет, то есть `СОЗДАТЬ РК`.

В сообщении:

```text
↳ РК: **ПРОВЕРИТЬ АКТИВНОСТЬ РК** (*траты 3д: 1 ₽*) / @marketer
```

В Markdown-таблице действие пишется без капса:

```text
РК: проверить активность РК
```

## Формат сообщения в Пачке

Сообщение всегда начинается:

```text
**ПОСТАВКИ FBO WB**
```

Далее идут блоки:

```text
**ВЧЕРА и СЕГОДНЯ (проверить цену / включить РК / включить БЗО):**
`333` (`+336`) / `679796751` / Коврик детский 2.3 складной 150x200 город
↳ РК: **ПРОВЕРИТЬ АКТИВНОСТЬ РК** (*траты 3д: 1 ₽*) / @a.nekrasov

**ЗАВТРА (проверить цену):**
`0` -> `+2370` / `224652683` / Сачок для бассейна 1.1 - синий
↳ РК: **СОЗДАТЬ РК** (*траты 3д: 19 ₽*) / @a.beaver
```

Артикул WB в сообщении оборачивается в backticks, чтобы его было удобно копировать.

## Формат Markdown-файла

Файл создается как:

```text
pachca_fbo_supplies_sheet_YYYY-MM-DD.md
```

Внутренние секции:

1. `ТОВАРЫ ИЗ ТЕКСТОВОЙ СВОДКИ` - таблицы только по товарам из сообщения.
2. `ЗАПРОС НА ДОБАВЛЕНИЕ ОТВЕТСТВЕННЫХ МАРКЕТОЛОГОВ` - если есть товары без маркетолога.
3. `ПОЛНЫЙ СПИСОК ПОСТАВОК` - все строки по источникам за 3 дня.
4. Таблицы по дням: `Сегодня`, `Завтра`, `Послезавтра`.

Текущие колонки таблиц:

- `Поставка`
- `Артикул ВБ`
- `Название товара`
- `Категория`
- `Действие`
- `Наличие кампании РК`
- `Траты (3д)`
- `Отзывы и рейтинг`
- `БЗО`
- `Заказы (7д)`
- `Цена / СПП`
- `Менеджер`
- `Маркетолог`

Если меняются колонки, нужно синхронно менять:

- заголовок таблицы полного списка;
- separator row;
- строку данных полного списка;
- заголовок таблицы `ТОВАРЫ ИЗ ТЕКСТОВОЙ СВОДКИ`;
- separator row;
- строку данных `ТОВАРЫ ИЗ ТЕКСТОВОЙ СВОДКИ`.

Все эти места сейчас находятся в `build_sheet_supplies_md.py`.

## Отправка в Пачку

`send_pachca_report.py` делает следующее:

1. Запускает `build_sheet_supplies_md.py`.
2. Загружает Markdown-файл через Pachca `/uploads`.
3. Отправляет основное сообщение в `entity_type=discussion` с `entity_id=PACHCA_CHAT_ID`.
4. Прикрепляет Markdown-файл к основному сообщению.
5. Если есть товары без маркетолога, создает тред к основному сообщению.
6. Отправляет запрос на добавление маркетологов в этот тред.

Если `thread_message` пустой, тред не создается.

## Что менять при добавлении новых данных

### Новый источник Google Sheets

1. Добавить URL/id/range в верхнюю часть `build_sheet_supplies_md.py`.
2. Если это поставки, обновить `SUPPLY_SHEET_SOURCES`.
3. Если это справочник, добавить отдельную функцию загрузки по аналогии с `load_marketer_by_article()` или `load_manager_by_article()`.
4. Если источник критичный, оставлять strict-поведение: при ошибке сборка должна падать, а не отправлять частичный отчет.

### Новый показатель из ABCAGE Analyzer

1. Добавить SQL-функцию по аналогии с:
   - `load_rk_by_article()`;
   - `load_ad_spend_by_article()`;
   - `load_orders_7d_by_article()`;
   - `load_price_by_article()`.
2. Вызвать функцию после формирования `stock_skus`.
3. Добавить поле в `grouped` default.
4. Заполнить поле в цикле по `source_rows`.
5. Добавить отображение в Markdown или сообщение.

### Новая рекомендация

1. Добавить функцию условия, например `needs_new_action(info)`.
2. Добавить текст в `action_label()` для Markdown-таблиц.
3. Добавить текст в `message_recommendations()` для сообщения в Пачку.
4. Проверить порядок рекомендаций. Сейчас порядок фиксирован: цена -> БЗО -> РК.
5. Если рекомендация должна работать только для блока `ВЧЕРА и СЕГОДНЯ`, использовать `include_rk_activity` или отдельный offset-флаг.

### Новый столбец в Markdown

Менять нужно обе таблицы:

- `ТОВАРЫ ИЗ ТЕКСТОВОЙ СВОДКИ`;
- дневные таблицы полного списка.

Не добавлять визуальные вертикальные разделители через отдельные `|`-колонки: в Пачке это ломает предпросмотр и делает таблицу слишком широкой.

## Безопасность

- Не коммитить токены, JSON service account, экспортированные env-файлы и сгенерированные отчеты.
- `GOOGLE_SERVICE_ACCOUNT_JSON` хранить только в GitHub Secrets или локальном env.
- `PACHCA_TOKEN` хранить только в GitHub Secrets или локальном env.
- `ABCAGE_ANALYZER_TOKEN` хранить только в GitHub Secrets или локальном `~/.codex/config.toml`.
- Не выводить секреты в логи.

## Возможный перенос в Cloudflare

Cloudflare Workers лучше подходит для постоянного cron-бота, но текущий проект нельзя просто скопировать туда как есть.

Причины:

- код написан на Python;
- используется subprocess;
- MCP подключается через `npx mcp-remote`;
- Cloudflare Workers не запускают локальные процессы и `npx`.

Варианты переноса:

1. Переписать сборщик на TypeScript Worker и обращаться к MCP/analyzer как к HTTP/JSON-RPC endpoint без `npx`.
2. Вынести Google Sheets в Apps Script, который будет передавать данные в Cloudflare KV/D1, а Worker будет собирать WB/MCP/Pachca-часть.
3. Оставить GitHub Actions как рабочий вариант, а Cloudflare сделать позже как отдельный runtime.

Если переносить, сначала нужно декомпозировать текущие функции Python на независимые слои:

- загрузка источников;
